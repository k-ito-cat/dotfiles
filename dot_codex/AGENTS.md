- 返答は日本語。
- 指示されたファイルや用語が見当たらない場合は、大小文字の違い・タイポ・名前の勘違いを疑って探索してから報告する。
- push/pull などの Git 操作や破壊的な操作は、明示的な指示があるまで行わない。

## コミット規約
### 対象になる変更
- ファイル分割、ディレクトリ移動、リネーム
- import の整理、設定ファイルの整理、定義位置の移動
- フォーマット適用、リンタの自動修正
- 実行結果が同じままの置換や整形

## コミットメッセージ規約

- `gcm`（generate commit message）: 以下の規約をもとにコミットメッセージの提案を行う。
- `gc!`（generate commit）: 以下の規約をもとにコミットメッセージの提案を行い、コミットの許可を確認する前に、対象コミットファイル（必要に応じてコミット行）と対応するコミット文章をリスト化して提示したうえで、許可を得たのちコミットする。

### 最優先事項

コミットメッセージを考える前に、まずコミット分割を検討する。

1. 現在の差分を全体として把握する
2. ファイル単位だけでなく、コード行単位で変更内容のまとまりを切り分ける
   - コミット対象ファイル内の差分に複数の変更が入り混じっている場合は、ファイル単位ではなくコード行単位でのコミット分割を検討する
   - 変更が多く複雑に入り混じっていて行単位での分割が現実的でない場合は、ファイル単位でコミットしてよい。その場合、コミットメッセージは具体を並べるのではなく、変更の共通意図を抽象化して表現してよい
3. 変更の性質が混ざっている場合は、コミットを分けることを最優先で検討する
   - 例: 同じファイル内に新規機能追加とスタイル変更が混在している場合、feat と style でコミットを分ける
   - 例: リファクタとバグ修正が混在している場合、refactor と fix でコミットを分ける
4. 大きい変更は段階的に分ける
   - 移行や刷新は一発でまとめず、動作を壊さない小さな差分に刻む
   - 途中段階でもテストやビルドが通る粒度を目安にする

この分割が完了したら、以下の規約に従ってコミットメッセージを書く。

### 形式

<type>[(scope)][!]: <subject>

- scope は必要であれば付ける
- 破壊的変更は `!` か `BREAKING CHANGE:` を使う

### type 一覧

- feat: 機能追加
- fix: バグ修正
- docs: ドキュメントのみ
- style: 見た目の変更のみ、挙動変更なし
- refactor: 機能追加でもバグ修正でもないコード変更
- perf: パフォーマンス改善
- test: テストの追加/修正
- build: ビルドや依存関係、ツール周り
- ci: CI設定やスクリプト
- chore: 雑務的な保守作業
- revert: 取り消し

### scope のルール

- 必要な場合だけ付ける
- できるだけ短く、対象領域が分かる名前にする
- 例: auth, api, ui, deps, build, ci, docs

例:
- feat(auth): ログインフォームにバリデーションを追加する
- fix(api): トークン未設定時にクラッシュしないようにする

### subject ルール

- 日本語で簡潔に書く
- 句点は付けない
- なるべく短く、目安は72文字以内
- 作業内容ではなく、変更内容または変更意図を明記する

悪い例:
- fix: バグ修正した
- feat: 対応
- chore: いろいろ

良い例:
- fix: トークン未設定時にクラッシュしないようにする
- feat: ログインフォームにバリデーションを追加する
- refactor: キャッシュキー生成を簡素化して重複を減らす
- style: ボタンの余白とフォントサイズを統一する

### 破壊的変更

どちらかで表現する:
- `type(scope)!: subject`
- footer に `BREAKING CHANGE: <説明>` を書く

例:
- feat(auth)!: 旧認証エンドポイントを削除する
- feat(api): 認証APIを刷新する
  BREAKING CHANGE: v1のエンドポイントを廃止し、v2に統一した

### squash する場合

WIPコミットが多い場合は squash して、最終コミットメッセージをこの規約に合わせる。

## コーディング規約
- コメントは追加しない。
- 既存のコメントは削除せず、維持する。

## 変更前の確認
- 機能追加、仕様変更、挙動変更、バグ修正を混ぜない
- ついでのロジック改善や最適化をしない
- リポジトリのREADME, package.jsonの既存スクリプトを見て、既に導入されている作法に合わせる。

## 実行すべきコマンド
- 変更後は、可能な範囲で以下を実行して結果を確認する（存在しない場合は無理に追加しない）。
  - フォーマット: `npm run format`
  - リント: `npm run lint`
  - テスト: `npm run test`

## 変更が大きい場合の追加確認
- 実装が複雑、影響範囲が広い、依存やビルド周りに触れた場合は、`npm run build`も実行して確認する。

## MCP
- Notion MCPを使う操作は、実行前に毎回ユーザーへ確認し、明示的な `yes` / `OK` がある場合のみ実行する。
- Notionのページ本文を更新する前に、`notion://docs/enhanced-markdown-spec` を確認し、対象ブロック（特に `columns`）の書式要件を先に確定する。
- `notion-update-page` は `data` を文字列ではなくオブジェクトで渡す。
- `columns` 内の本文更新では、列内要素のインデント（タブ）を維持した形式で更新する。
- 原文がメンション形式（例: `<mention-date .../>`）の場合、更新後もメンション形式を維持する。文字列日付へは変換しない。
- 大きな本文更新は一括反映しない。まず最小差分で反映テストし、`fetch` で崩れがないことを確認してから全量反映する。
- 本文更新の直後は必ず `fetch` で反映結果を確認し、欠落や崩れがあれば即座に段階的にリカバリする。

## navi運用方針

- コマンドリストを `navi` 以外の設定ファイル（`zsh` 定義の `alias` / `function` / `keybind` など）で管理している場合、`navi` には重複記載せず、ファイルパスを `cat` で参照するコマンドとして明記する。

## zsh運用方針

- `zsh` 設定は種別ごとにファイル分割して管理する。
- 例:
  - `core` 系: 基本設定・環境初期化
  - `aliases` 系: alias
  - `functions` 系: function / widget 初期化
  - `keybinds` 系: bindkey
- 新しい定義を追加・修正する場合は、まず適切な種別のファイルに配置する。
- `navi` 側には必要に応じて参照導線のみ追加し、実体の重複は避ける。

## chezmoi運用方針

- `chezmoi` は、機密情報を含むファイルを除く `dot config` のバージョン管理と再現性担保を目的として運用している。
- `dot config` が `chezmoi` 管理対象の場合、更新は基本的に `chezmoi` 側の `tmpl` ファイルを優先する。
- 参照先としても、生成先の実ファイルより `chezmoi` 管理元（`tmpl` 側）を優先する。
- 実ファイルを確認する場合でも、真実源は `chezmoi` 管理元として扱う。
